# 数值系统改造方案 V1

状态：推荐基线；Slice 1 已于 2026-07-20 实现并通过自动测试，Slice 2–4 待实施<br>
范围：精力、恢复、金币、成就、日期结算与相互关系<br>
方法：`game-economy-planning` 技能的资源语义、来源/消耗、场景模拟、成就可达性和刷取审计

## 1. 产品目标

数值系统服务于“在有限精力里做可承受的取舍”，而不是鼓励用户刷任务或在低精力时继续硬撑。

目标闭环：

```text
识别现实容量
→ 选择或调整任务
→ 完成后结算精力与金币
→ 必要时安排真实恢复
→ 记录可验证行为
→ 解锁成就并自动发放奖励
→ 形成下一次更可靠的决策依据
```

非目标：

- 不把金币做成真实货币或付费点；
- 不用精力阻止用户如实记录已经完成的现实任务；
- 不奖励熬夜、透支、低精力清空全部任务等不健康行为；
- 不让恢复任务成为刷金币或刷普通任务数的最优策略。

## 2. 资源定义

| 资源 | 定义 | 权威来源 | 时间规则 | 上限 |
| --- | --- | --- | --- | ---: |
| 精力 | 用户当天可支配的行动预算；100 为标准基线，101–150 为充盈缓冲 | 系统结算，用户每日校准 | 跨日自然恢复，任务消耗，休整恢复 | 150 |
| 金币 | 对有效执行和取舍的游戏化反馈货币 | 普通任务、成就自动奖励 | 当前余额与历史累计统一结算；未来可用于非必要增强和装饰 | 无硬上限 |
| 历史金币累计 | 任务与成就实际到账金币的累计值 | 统一结算账本 | 长期累计，不因消费回退 | 无 |
| 实际恢复累计 | 在未满精力时真正恢复的数值 | 恢复结算后的实际增量 | 长期累计；每日最多 60 点计入成就 | 无 |
| 成就进度 | 对具体行为或长期状态的验证 | 统一领域事件 | 长期保留 | 各成就独立 |

用户校准是现实状态修正，不是恢复来源；将精力从 20 校准到 100，不增加“实际恢复累计”。

## 3. 精力数值

### 3.1 普通任务

| 预计时长 | 档位 | 精力消耗 | 金币奖励 |
| --- | --- | ---: | ---: |
| 5–30 分钟 | LIGHT | 10 | 10 |
| 31–90 分钟 | MEDIUM | 20 | 20 |
| 91–240 分钟 | HEAVY | 30 | 30 |

选择依据：

- 以 100 为标准日预算并保留 50 点充盈缓冲，避免当前 1/2/3 与 100/150 刻度不在同一量级；
- 25 分钟心流消耗 10，与轻任务一致；
- 典型日“两个轻任务 + 一个中任务”消耗 40，给用户保留现实波动和临时事务余量；
- 高负荷日“一轻 + 两中 + 一重”消耗 80，需要恢复或次日调整。

### 3.2 跨日与校准

- 新用户初始精力：100。
- 自然跨日恢复：+30，最高到 150，不计入成就恢复累计。
- 每个自然日首次进入应用时提示校准，可选择滑杆或快捷档：低 30、中 70、高 100、充盈 130。
- 每日允许再次校正，但必须记录 `ENERGY_CALIBRATED`，不产生恢复成就。
- 日期以本地 `YYYY-MM-DD` 为日界，重复启动或刷新不得重复发放跨日恢复。

### 3.3 容量约束

- 生成或批准任务时，如果预计总消耗超过当前精力，必须显示缺口并提供：拆小、延期、先恢复、明确透支四种选择。
- 对尚未开始的任务，精力不足时不允许无提示地继续接受重任务。
- 对现实中已经完成的任务，允许如实标记完成；精力降到 0，并记录一次 `ENERGY_OVERDRAWN`，不能出现负数。
- 精力为 0 时始终保留免费恢复入口，避免死锁。

## 4. 恢复系统

### 4.1 恢复档位

恢复使用独立档位，不再单纯复用普通任务时长档位：

| 恢复档位 | 建议行为 | 时长 | 恢复精力 | 金币 | 是否计入普通任务数 |
| --- | --- | ---: | ---: | ---: | --- |
| 微休整 | 离屏、呼吸、补水、伸展 | 10–15 分钟 | +10 | 0 | 否 |
| 标准休整 | 散步、用餐、安静休息 | 30–45 分钟 | +20 | 0 | 否 |
| 深度休整 | 午睡、运动后放松、完整离线 | 60–90 分钟 | +30 | 0 | 否 |

结算使用实际增量：精力 85 时完成标准休整，只恢复 15，累计恢复也只增加 15。

### 4.2 触发矩阵

| 触发来源 | 条件 | 系统行为 | 是否自动生成任务 |
| --- | --- | --- | --- |
| 主动入口 | 精力低于 150 | 钦天监显示三种休整选项 | 用户确认后生成 |
| 低精力提示 | 精力 ≤30 | 顶栏和大臣提示先恢复或降级 | 否 |
| 临界状态 | 精力 ≤10 | 暂停推荐重任务，突出免费微休整 | 否 |
| 对话语义 | 用户表达累、熬夜、撑不住、需要休息 | AI 可生成 `recovery` 任务 | 用户朱批后生成 |
| 连续专注 | 单次 ≥90 分钟或连续两个心流段 | 轻提示微休整 | 否 |
| 透支事件 | 任务完成导致精力归零 | 将恢复列为下一推荐动作 | 否 |
| 跨日 | 日期变化 | 自然恢复 +30 | 不生成任务 |

进入钦天监本身不应再是空场景：即使没有 AI 任务，也要显示主动休整入口和当前可恢复量。

### 4.3 防刷规则

- 恢复任务不发金币；
- 恢复任务不计入瓷器普通任务完成数；
- 满精力时可以查看恢复建议，但不能结算正数；
- 累计恢复只记实际增量；
- 每日最多 60 点实际恢复计入累计恢复成就，额外恢复仍可改变当前精力；
- 同一恢复任务只能结算一次。

## 5. 金币经济

### 5.1 来源

| 来源 | 数值 | 是否计入历史金币累计 |
| --- | ---: | --- |
| LIGHT 普通任务 | 10 | 是 |
| MEDIUM 普通任务 | 20 | 是 |
| HEAVY 普通任务 | 30 | 是 |
| 恢复任务 | 0 | 否 |
| 成就自动奖励 | 按成就定义 | 是 |
| 演示/迁移补偿 | 单独来源 | 否 |

金币成就的 50/100/300/500/1000/3000 阈值统一读取历史金币累计。成就奖励与任务奖励属于同一种金币，因此允许成就到账继续推动财富里程碑；成就 ID 和结算 ID 保证连锁只发生一次。

典型用户每天约获得 40 金，因此预计：

| 阈值 | 预计到达时间 |
| ---: | --- |
| 50 | 第 2 天 |
| 100 | 第 3 天 |
| 300 | 第 8 天 |
| 500 | 第 13 天 |
| 1000 | 第 25 天 |
| 3000 | 第 75 天 |

### 5.2 消耗

核心决策、低精力建议和免费恢复不能收费。金币只购买非必要增强：

| 消耗项 | 建议价格 |
| --- | ---: |
| 追加一个不同风格的大臣视角 | 50 |
| 对同一决策追加一次预言推演 | 30 |
| 普通称号、头像框或御案装饰 | 200 |
| 稀有装饰套件 | 500 |

消费功能未实现前，界面应明确写“金币当前作为功绩积分累计”，并暂时隐藏“首次消费”成就，避免承诺不存在的用途。

## 6. 成就奖励

成就奖励在解锁时自动进入统一结算：

| 等级 | 默认奖励 |
| --- | --- |
| Tier 1 | 10 金 |
| Tier 2 | 20 金 |
| Tier 3 | 30 金 |
| Tier 4 | 50 金或稀有装饰 |
| Tier 5 | 称号/限定装饰 + 100 金 |

成就状态：

```js
{
  unlocked: false,
  progress: 0,
  unlockedAt: null,
  rewardGranted: false,
  rewardGrantedAt: null
}
```

自动发放使用 `achievement:<achievement_id>` 作为幂等结算键；刷新、重复事件或旧存档再次加载不得重复发放，不提供“待领取”状态。

### 7.3 玉雕·精力

### 7.4 金器·金币

- 历史阈值读取任务与成就统一的历史金币累计；
- “单次 100 金”改为首次完成 HEAVY 任务；
- “连续三日进账”使用每日任务金币快照；
- “单日 200 金”改为单日任务金币 100，符合每天 1–4 项任务的产品约束；
- “首次消费”在金币消耗上线后开放；
- “金精两全”改为当前金币 ≥500 且精力 ≥100；
- 恢复任务不能贡献金币来源多样性。

### 7.5 木器·其他

- “迷途知返”增加 `folk → court → TASK_COMPLETED` 的顺序状态；
- “三思后行”必须记录同一个 decision ID 的三次推演；
- 其余地图、藏书、心流、预言和朱批成就保留。

## 8. 数值模拟

模型：30 天；目标方案包含跨日自然恢复 +30、精力上限 150、恢复无金币。模拟初始状态不结算跨日恢复，从第 2 天开始结算。下表金币只表示可重复的任务基础收入，尚未叠加一次性成就奖励；实际统一金币累计会更高。

| 场景 | 当前规则 30 日 | 目标规则 30 日 | 判断 |
| --- | --- | --- | --- |
| 低负荷：1 轻 + 1 微休整/日 | 精力 150，金币 600 | 精力 150，金币 300 | 低负荷可积累充盈缓冲，无额外刷金 |
| 典型：2 轻 + 1 中 + 1 微休整/日 | 精力 150，金币 1500 | 精力 70，金币 1200 | 目标保留余量，数值能表达负荷 |
| 高负荷：1 轻 + 2 中 + 1 重 + 1 标准休整/日 | 精力 150，金币 3000 | 精力 40，29 项因容量不足被提示调整 | 目标能识别长期过载 |
| 刷恢复：10 次微休整/日 | 精力 150，金币 3000 | 精力 150，金币 0，仅首日实际恢复 50 | 目标消除持续正收益刷取 |

模拟配置位于 `docs/数值模拟配置-V1.json`，模拟器位于项目根目录的 `skills/game-economy-planning/scripts/simulate_economy.mjs`。

## 10. 旧存档迁移

- `energy = min(150, oldEnergy)`；
- `energyCap = 150`；
- 当前金币和历史累计金币保留，旧 `totalGold` 直接作为统一历史累计；
- 新的实际恢复累计从 0 开始，避免旧版刷恢复污染新阈值；
- 已解锁但从未发奖的旧成就迁移后补发一次，并写入 `achievement:<id>` 结算记录；
- 未解锁成就按新规则重新计算进度；
- 已完成任务可以按 `cat !== recovery/mystic` 重建普通任务数；
- `startedAt` 转换为 ISO；转换失败时保留当前 `day`，并以迁移当天建立 `dayKey`。

## 12. 自动验收

必须覆盖：

1. LIGHT/MEDIUM/HEAVY 分别结算 10/20/30 精力和金币；
2. 重复完成同一任务不重复扣精力、加金币或增加任务数；
3. 精力不足的任务在批准前出现拆分/延期/恢复/透支选择；
4. 已经完成的现实任务仍可记录，精力最低为 0 并产生透支事件；
5. 恢复任务不发金币、不计普通任务数；
6. 恢复只统计上限以内的实际增量；
7. 满精力重复恢复不会增加累计恢复；
8. 跨日 +30 每日只结算一次；
9. 校准不增加累计恢复；
10. 金币成就读取任务与成就统一的历史累计金币；
11. 同一成就奖励在重复事件和重复加载下只发放一次；
12. 成就奖励可以推动财富成就，但每个连锁节点只能结算一次；
13. 时间连续类成就使用真实日快照；
14. 复合成就缺少任一条件时不解锁；
15. 旧 V1 存档迁移后资源、任务和已解锁展示不丢失。

## 13. 上线后观测

首轮收集但不用于个体绩效评判：

- 每日精力校准分布；
- 任务接受时的预计精力缺口；
- 各档任务完成率和延期率；
- 低精力提示后的拆分/延期/恢复选择；
- 恢复入口曝光、开始和完成率；
- 日均任务金币收入与消费；
- 成就解锁和自动到账失败率；
- 精力归零和透支事件频率。

两周后再调整数值。若典型用户日末中位精力长期高于 100，说明成本仍太轻；长期低于 40，说明成本过重或每日自然恢复不足。第一版目标区间为典型用户日末 55–85。
