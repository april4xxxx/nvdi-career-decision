# 女皇入朝：AI 编排与输出契约 V2.2

版本：V2.2<br>
日期：2026-07-19<br>
上位业务规则：[产品 PRD V2.2](01-产品PRD-V2.2.md)

## 1. 文档目的

本文档定义 AI 如何读取用户画像、分类问题、检查信息、检索 SOP、生成奏折、处理调整、形成任务、复盘记忆和执行预言模式。

本文档不定义：

- UI 布局、地图和视觉；
- 大臣具体文风；
- SOP 的真实内容；
- 金币和精力的具体数值配置；
- 御前推演六道题目的文案。

## 2. 事实源优先级

发生冲突时按以下顺序处理：

```text
安全与合规规则
> 用户本轮明确事实和目标
> 公司正式制度 / 用户提供的正式文件
> 已确认且近期的用户画像与任务记录
> 当前阶段匹配的运行时 SOP
> 一般职场方法
> AI 推断
> 大臣角色文风
```

角色文风永远不能覆盖结构化字段。

## 3. 输入契约

### 3.1 用户上下文

```json
{
  "user_id": "user_001",
  "career_phase": "posthire",
  "journey_stage": "DAY_8_30",
  "active_goal": "完成第一次独立交付",
  "availability": [],
  "current_commitments": [],
  "protected_items": [],
  "preferred_task_size_minutes": 30,
  "decision_profile": {},
  "confirmed_memories": [],
  "recent_task_summary": {},
  "role_style_id": "ROLE_PENDING"
}
```

### 3.2 本轮输入

```json
{
  "user_input": "老板让我周五做一次行业分享，我要不要认真准备？",
  "conversation_summary": "",
  "clarification_count": 0,
  "existing_issue_id": null
}
```

### 3.3 御前推演画像接口

AI 可读取以下字段：

- `decision_approach`；
- `authority_response`；
- `conflict_style`；
- `initiative_style`；
- `feedback_orientation`；
- `decision_reef_candidate`；
- `confidence`。

使用限制：

1. 画像只调整追问、证据、提醒和任务颗粒度；
2. `confidence=initial` 时必须使用“候选偏好”，不能当成稳定事实；
3. 用户本轮表达优先于历史画像；
4. 画像不能独立改变风险、阶段、决策终态和 SOP 适用性；
5. 只有用户确认的长期记忆可以稳定用于后续适配。

## 4. 总编排流程

```text
0. 输入校验
→ 1. 职业阶段与范围路由
→ 2. 主题、意图与风险分类
→ 3. 问题解析与信息充分性判断
→ 4. 必要追问
→ 5. RESPOND_ONLY / EXECUTE / ARCHIVE 判断
→ 6. SOP 检索
→ 7. SOP 裁剪与任务生成
→ 8. 精力、时间和已有任务协调
→ 9. 生成结构化奏折
→ 10. 角色文风渲染
→ 11. 用户同意 / 再议 / 大胆
→ 12. 任务持久化与完成
→ 13. 复盘和候选记忆
```

预言模式使用独立只读流程，不能绕过普通任务确认。

## 5. 步骤 0：输入校验

调用 AI 前由程序校验：

- `career_phase` 和 `journey_stage` 是否为合法枚举；
- 用户画像、当前任务和时间字段是否符合 Schema；
- `clarification_count` 是否为 0-2；
- 当前阶段对应的 SOP 库是否可用；
- 是否包含不应发送给模型的敏感数据。

失败时返回程序错误，不要求模型猜测缺失参数。

## 6. 步骤 1：职业阶段与范围路由

### 6.1 阶段规则

- 保存阶段来自用户已确认选择；
- 用户明确说状态变化时，返回 `requires_phase_confirmation=true`；
- 用户确认后更新阶段并重新处理本轮问题；
- AI 不得擅自把求职用户改成入职后，或反向修改。

### 6.2 范围状态

```text
IN_SCOPE
OUT_OF_SCOPE
ESCALATE
```

- `IN_SCOPE`：继续普通流程；
- `OUT_OF_SCOPE`：说明产品边界，不检索 SOP、不生成任务；
- `ESCALATE`：进入专门帮助流程，不生成普通职场任务。

## 7. 步骤 2：分类

分类字段来自 [分类字典](03-分类字典.md)：

- `career_phase`；
- `journey_stage`；
- `primary_topic`；
- `secondary_topics`，最多 2 个；
- `primary_intent`；
- `secondary_intents`，最多 2 个；
- `risk_level`；
- `content_priority`。

主分类回答“用户本轮最想解决什么”，不能按关键词数量投票。

### 7.1 分类输出

```json
{
  "career_phase": "posthire",
  "journey_stage": "DAY_8_30",
  "primary_topic": "workload_priority",
  "secondary_topics": ["manager_communication"],
  "primary_intent": "decide",
  "secondary_intents": ["plan"],
  "risk_level": "normal",
  "content_priority": "core",
  "needs_retrieval": true,
  "retrieval_library": "workplace_newcomer_year_one",
  "confidence": 0.91,
  "reason": "用户需要决定额外任务的投入程度，并处理与已有任务的冲突"
}
```

## 8. 步骤 3：问题解析

必须提取：

- 用户当前问题；
- 用户希望获得的结果；
- 明确期限和紧迫度；
- 用户可控制的动作；
- 是否属于明确职责或已有承诺；
- 初步时间和精力成本；
- 会受影响的已有任务或不可牺牲项；
- 缺失信息；
- 安全和范围标记。

```json
{
  "issue": "行业分享应该投入多少准备时间",
  "desired_outcome": "完成必须的分享，同时避免挤掉重要安排",
  "deadline": "周五",
  "user_controllable_action": "限制准备范围和时间",
  "obligation": "REQUIRED_BUT_SCOPABLE",
  "estimated_cost": {
    "minutes": 90,
    "energy_tier": "MEDIUM"
  },
  "affected_commitments": [],
  "missing_information": ["分享的最低交付要求"],
  "information_sufficient": false,
  "safety_flag": "NONE"
}
```

## 9. 步骤 4：必要追问

只有缺失信息可能改变以下任一内容时才追问：

- `IN_SCOPE / OUT_OF_SCOPE / ESCALATE`；
- `RESPOND_ONLY / EXECUTE / ARCHIVE`；
- 风险等级；
- SOP 选择；
- 任务投入上限。

规则：

- 一次只问一个问题；
- 最多 2 轮；
- 提供 2-4 个简短选项并允许自由输入；
- 用户不知道或不想回答时停止；
- 剩余未知不会改变结果时停止；
- 角色文风可以改变问法，不能改变问题语义。

## 10. 步骤 5：普通处理结果

普通事项只允许：

- `EXECUTE`；
- `ARCHIVE`；
- `RESPOND_ONLY`。

### 10.1 EXECUTE 条件

通常满足至少一项：

- 明确职责、承诺或必要风险处理；
- 直接服务当前阶段目标；
- 有清晰、用户可控且成本可承受的动作；
- 通过小成本行动可以获得会改变后续选择的信息。

### 10.2 ARCHIVE 条件

通常满足至少一项：

- 与当前目标无关且会挤占重要安排；
- 用户无法控制结果，也没有可行的最小动作；
- 只是因为焦虑、讨好或金币而新增任务；
- 当前信息显示投入明显大于合理价值；
- 已有等效安排，无需重复任务。

### 10.3 RESPOND_ONLY 条件

通常满足以下全部条件：

- 用户本轮只需要概念解释、事实澄清、情绪承接或简单信息；
- 没有要求形成计划、决定、话术、模板或下一步；
- 当前回答不需要创建、替换、改期或归档任务；
- 不属于 `OUT_OF_SCOPE` 或 `ESCALATE`。

`RESPOND_ONLY` 不检索 SOP，不生成任务，不进入任务确认。用户随后明确要求行动时，重新进入 `EXECUTE | ARCHIVE` 判断。

### 10.4 判断输出

```json
{
  "decision": "EXECUTE",
  "reason_codes": ["ROLE_OBLIGATION", "COST_CAN_BE_BOUNDED"],
  "reason_summary": "分享必须完成，但准备程度可以控制",
  "execution_constraints": {
    "max_minutes": 90,
    "max_energy_tier": "MEDIUM",
    "protected_items": ["周五晚休息"]
  }
}
```

`RESPOND_ONLY` 输出：

```json
{
  "decision": "RESPOND_ONLY",
  "reason_codes": ["NO_ACTION_REQUIRED"],
  "reason_summary": "用户本轮只需要解释，不需要形成任务",
  "direct_response": "",
  "requires_user_confirmation": false
}
```

## 11. 步骤 6：SOP 检索

### 11.1 库选择

- `career_phase=job_seeking`：只检索 `workplace_job_seeking`；
- `career_phase=preboarding`：优先检索求职库中的 preboarding 模块；
- `career_phase=posthire`：只检索 `workplace_newcomer_year_one`；
- 事件型 topic 只有用户主动提出时可检索；
- 风险升级状态不检索普通 SOP。
- 只有 `decision=EXECUTE` 才检索 SOP；`ARCHIVE` 与 `RESPOND_ONLY` 均直接跳过。

### 11.2 检索条件

- 阶段兼容；
- 主主题和必要辅助主题；
- 用户目标；
- 适用条件；
- 排除条件；
- 风险限制；
- 时间和精力容量。

### 11.3 检索输出

```json
{
  "retrieval_status": "MATCHED",
  "library_id": "workplace_newcomer_year_one",
  "query": "入职早期必须行业分享但准备容易过度，控制投入并保护已有任务",
  "required_tags": ["posthire", "workload_priority", "performance_visibility"],
  "candidates": [
    {
      "sop_id": "SOP_ID_PLACEHOLDER",
      "version": "",
      "match_reason": "",
      "excluded_steps": []
    }
  ]
}
```

无匹配时：

```json
{
  "retrieval_status": "NO_MATCH",
  "library_id": "workplace_newcomer_year_one",
  "candidates": []
}
```

## 12. 步骤 7：SOP 裁剪与任务生成

规则：

1. SOP 是候选模板，不要求全部执行；
2. 删除不适用当前阶段、岗位和时间的步骤；
3. 只保留服务本轮目标的最小步骤；
4. 每步必须由用户本人控制；
5. 等待他人回复不能作为用户的完成标准；
6. 默认 1-3 步，复杂事项最多 5 步；
7. 每步包含动作、时长、建议窗口和完成标准；
8. 输出来源为 `SOP | GENERATED | HYBRID`；
9. `GENERATED` 只允许补充最少量通用步骤；
10. 角色文风不能修改任务字段。

## 13. 步骤 8：任务协调与精力预占

### 13.1 精力状态

```json
{
  "period": "2026-W29",
  "capacity_total": 10,
  "capacity_available": 3,
  "capacity_reserved": 5,
  "capacity_spent": 2
}
```

### 13.2 协调规则

- 候选方案不预占精力；
- 用户同意后才预占；
- 超过容量时必须提出 `REPLACE | COMPRESS | RESCHEDULE`；
- 不得静默删除已有任务；
- 用户不可牺牲项优先于普通候选任务；
- 完成后将 `reserved` 转到 `spent`，不再次减少可用容量；
- 取消或归档释放预占。

## 14. 步骤 9：结构化奏折

```json
{
  "issue_id": "issue_001",
  "scope_status": "IN_SCOPE",
  "decision": "EXECUTE",
  "title": "行业分享·足够好版本",
  "reason_codes": ["ROLE_OBLIGATION", "COST_CAN_BE_BOUNDED"],
  "known_facts": [],
  "inferences": [],
  "execution_constraints": {
    "max_minutes": 90,
    "energy_tier": "MEDIUM"
  },
  "affected_commitments": [],
  "plan_source": "HYBRID",
  "matched_sop_ids": [],
  "tasks": [
    {
      "task_id": "candidate_001",
      "action": "列出听众最关心的3个问题",
      "duration_minutes": 15,
      "suggested_window": "周四20:00",
      "done_criteria": "已写出3个问题",
      "energy_tier": "LIGHT"
    }
  ],
  "requires_user_confirmation": true
}
```

当 `decision=RESPOND_ONLY` 时：

- `plan_source`、`matched_sop_ids` 和 `tasks` 固定为空；
- `requires_user_confirmation=false`；
- 输出 `direct_response`，不得伪造任务、精力或金币；
- 用户追问如何行动时，再重新分类并进入普通判断。

## 15. 步骤 10：角色文风渲染

渲染器输入结构化奏折，输出面向用户的角色文本。

允许变化：

- 称呼；
- 句式；
- 长短；
- 幽默和比喻；
- 直接程度；
- 宫廷世界观词汇。

禁止变化：

- 所有枚举；
- 已知事实和推断边界；
- 风险等级；
- 判断终态；
- SOP ID；
- 任务动作、时长和完成标准；
- 精力与金币；
- 是否需要用户确认。

角色规范确定后，应增加角色一致性评测集。

## 16. 步骤 11：用户操作

### 16.1 同意

- 校验候选任务仍未过期；
- 再次检查容量；
- 写入任务池；
- 预占精力；
- 记录用户确认时间。

### 16.2 再议

可表达：

- 时间太长；
- 难度太高；
- 拆得更细；
- 改到其他时间；
- 与已有任务冲突；
- 用户补充了新事实。

若新事实改变目标或必要性，重新进入判断步骤。

### 16.3 大胆

- 记录 `USER_CHALLENGED_JUDGMENT`；
- AI 不辩解、不假装确定；
- 说明当前判断依赖了哪些假设；
- 只追问一个最能改变判断的问题；
- 重新生成结构化结果。

## 17. 完成、复盘与记忆

任务状态：

```text
PENDING | IN_PROGRESS | COMPLETED | PARTIAL | NOT_DONE | CANCELED
```

完成后：

1. 记录实际时长和用户反馈；
2. 精力从预占转为已消耗；
3. 固定规则计算金币；
4. 生成一句与真实结果一致的反馈；
5. 提取候选记忆；
6. 用户选择 `REMEMBER | THIS_TIME_ONLY | DISCARD`；
7. 只有 `REMEMBER` 可以写入长期记忆。

一次行为不能升级为永久偏好。御前推演画像的长期确认继续遵守现有“三次真实事件 + 用户确认”规则。

## 18. 预言模式契约

### 18.1 启动条件

- 用户主动打开；
- 存在已确认任务；
- 有足够数据支持对应时间范围；
- 不处于高风险升级流程。

### 18.2 输出结构

```json
{
  "data_status": "SUFFICIENT",
  "observation_window_days": 30,
  "facts": [],
  "baseline_scenario": {
    "horizon_days": 7,
    "assumptions": [],
    "likely_outcomes": [],
    "risks": []
  },
  "alternative_scenario": {
    "changed_assumption": "行业分享准备时间从4小时降至90分钟",
    "likely_outcomes": [],
    "tradeoffs": []
  },
  "confidence_level": "MEDIUM",
  "limitations": [],
  "proposed_changes": [],
  "requires_user_confirmation": true
}
```

### 18.3 不变量

- `facts` 必须可追溯到任务记录；
- 推断必须写明观察窗口；
- 情景必须写明假设；
- 数据不足返回 `INSUFFICIENT_DATA`；
- 不生成晋升、收入、转正或离职的确定结果；
- 不自动修改任务。

## 19. 错误与安全回退

| 错误 | 回退行为 |
| --- | --- |
| JSON 解析失败 | 重试一次；仍失败则返回固定安全错误，不执行写操作 |
| 非法枚举 | 拒绝结果并重新生成 |
| 阶段缺失 | 要求上游补齐，不由模型猜测 |
| SOP 库不可用 | 返回 `NO_MATCH`，不伪造 SOP ID |
| 任务超过容量 | 不允许确认，要求调整 |
| 风险不确定 | 使用更高风险等级并进入人工/专门流程 |
| 角色文本与结构化结果冲突 | 丢弃角色文本，展示中性结构化结果 |
| 预言依据不足 | 返回 `INSUFFICIENT_DATA` |

## 20. 全局禁止行为

- 虚构公司制度、同事动机或职业后果；
- 用角色人设合理化欺骗、操控、歧视或危险建议；
- 让角色语气弱化风险提醒；
- 把御前推演当成心理诊断或固定人格；
- 把等待他人认可作为用户任务；
- 为了金币增加任务；
- 归档后生成补救性社交任务；
- 静默删除、接受或改期用户任务；
- 使用未匹配的 SOP ID；
- 在数据不足时输出伪精确预测；
- 未经确认写入长期记忆。
